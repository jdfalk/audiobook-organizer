# `SLOG-PROD-VERIFY` Smoke Test

Automate the verification that a metadata-fetch operation emits the
full slog/activity chain on production: the operation ID is stamped in the
logs and the new `/api/v1/operations/:id/activity` endpoint returns the
activity rows generated by that `metadata-fetch` run.

## Preconditions

1. You can authenticate against production (API token or session cookie).
2. You can read the activity logs produced by the running service
   (journal, structured sink, or whatever log aggregator the prod
   deployment ships to).
3. The metadata-fetch operation has permission to run on a small test
   book (pick a test book that is safe to refresh).

## Steps

1. **Trigger a metadata fetch.**
   Use the UI or API to start a `metadata-fetch` operation for a known
   book (e.g. POST `/api/v1/audiobooks/<book-id>/metadata-fetch`).
   Record the returned operation ID from the response (look for
   `data.operation.id`, `op_id`, or `id`). We will refer to that value as
   `OPID`.

2. **Wait for the operation to emit logs.**
   Keep an eye on the running ticker / logs until the `metadata-fetch`
   job reaches completion. In the structured slog output you should
   eventually see entries tagged with `op:<OPID>` and `type` values
   like `metadata-fetch` / `metadata-apply`. The log lines should
   include `opID=<OPID>` so you can confirm the log stream has the
   target operation in it.

3. **Inspect the activity API.**
   Run a request against `GET /api/v1/operations/<OPID>/activity` while
   the operation is running and again after it completes. Each
   response should include a `data.activity` array with rows that came
   from the same op (start message, progress updates, completion). If
   the API returns an empty array while the operation is still running,
   wait a few seconds and try again; the new endpoint is backed by the
   activity store and is eventually consistent.

   Example command (replace `<BASE>` and `<OPID>`):

   ```bash
   curl -H "Authorization: Bearer $PROD_TOKEN" \
     "https://<BASE>/api/v1/operations/<OPID>/activity" | jq '.data.activity | length'
   ```

   The length should be `> 0` once the operation emits at least one
   info-tier activity entry. The returned rows should include
   `Tier: info`, `Type: metadata-fetch` / `metadata-apply`, `Source:
   metadata`, and the `op:<OPID>` `action:` tags described in the
   activity enrichment doc.

## Verification checklist

- [ ] `metadata-fetch` operation returned an explicit op ID and is
  visible in the structured log stream (`op=<OPID>` / `op:<OPID>`).
- [ ] `/api/v1/operations/<OPID>/activity` returns multiple rows for the
  same op (start/progress/completion). If the current operation is
  still streaming, expect the first request to eventually show rows.
- [ ] Activity rows include the metadata domain tags (e.g. `action:
  metadata-apply`, `domain:metadata`, `source:metadata`).
