# UOS-01 — Schema migrations for `*_v2` tables

**Companion human spec:** `docs/superpowers/specs/2026-05-04-unified-operations-system.md` §2

## Branch

```
feat/uos-01-schema
```

## Goal

Land all `*_v2` tables described in spec §2.1 as a single migration.
No code uses these tables yet — this PR is schema-only.

## Files to add

1. `internal/database/migrations/NNN_uos_schema_v2.sql` — pick the
   next available migration number. SQL EXACTLY as specified in spec
   §2.1, including index definitions. Insert one row into
   `core_schema_meta_v2` with `core_schema_version = 1`.
2. `internal/database/migrations/NNN_uos_schema_v2_test.go` — tests
   that:
   - Each table exists and has every column at the right type.
   - Each index exists.
   - The unique-id check on `core_schema_meta_v2` works (insert two
     rows; second insert MUST fail).
   - Down-migration drops every table and index cleanly.

## Files to edit

1. `internal/database/migrations/migrations.go` (or wherever the
   migration list lives; do `grep -n "Migration{" internal/database/migrations/`
   to find it). Append the new migration to the list.

## Hard rules

- Migration MUST be reversible. Provide both `Up` and `Down`.
- Column types MUST match spec §2.1 byte-for-byte. No "TIMESTAMP DEFAULT
  CURRENT_TIMESTAMP" sugar — the registry sets timestamps explicitly.
- All indexes named in §2.1 MUST be created.
- Do NOT modify the existing `operations` or `operation_logs` tables in
  this PR. They stay live for the dual-source phase.
- Do NOT seed any rows other than the single `core_schema_meta_v2`
  row.

## Acceptance criteria

- [ ] `go test ./internal/database/migrations/...` passes.
- [ ] `make build` passes.
- [ ] `make ci` passes.
- [ ] Manual: run the binary against a fresh DB, confirm all `*_v2`
      tables exist via `sqlite3 ... ".schema operations_v2"` and same
      for the other tables.
