# UOS-10 — Migrate iTunes plugin

**Companion human spec:** §11 phase B.

## Branch

```
feat/uos-10-itunes
```

## Goal

Migrate the iTunes-related operations to the UOS plugin contract.
Existing ops (from grep on the codebase):

- `itunes_import` — bulk import from iTunes XML
- `itunes_sync` — pull updates from iTunes
- `itunes_path_reconcile` — reconcile iTunes paths to library paths
- `itunes_path_repair` — repair broken iTunes paths
- `itunes-position-sync` — sync playback positions

## Files to add

1. `internal/plugins/itunes/plugin.go` — plugin shell. Migrations
   list MUST include the per-plugin tables iTunes already uses
   (these were core migrations historically; per spec §2.3 they stay
   in core but the iTunes plugin declares `RequiresCoreSchema:
   ">=1"`).

2. `internal/plugins/itunes/import.go` — `importDef()`:
   - `ID`: `"itunes.import"`
   - `Run`: extracted from `internal/itunes/service/importer.go` flow
     currently invoked by `s.itunesSvc.Importer.Execute`
   - `ResumePolicy`: `ResumeRestart` (long-running, valuable to
     resume) — Run uses `Reporter.Checkpoint` to persist
     progress every 1000 tracks
   - `Phases`: `[{Name: "parse_xml"}, {Name: "match_books"},
     {Name: "import_tracks"}, {Name: "post_process"}]`
   - `Cancellable`: `true`
   - `Capabilities`: `[CapLibraryRead, CapLibraryWrite,
     CapFilesRead]`
   - `ParamsSchema`: validates the existing `ITunesImportParams`
     struct
   - `Permissions`: existing `auth.PermScanTrigger`

3. `internal/plugins/itunes/sync.go` — `syncDef()`:
   - `ID`: `"itunes.sync"`
   - `ResumePolicy`: `ResumeRequeue`
   - `Schedule`: nightly (`"0 4 * * *"`)
   - `Capabilities`: same as import

4. `internal/plugins/itunes/path_reconcile.go` —
   `pathReconcileDef()`:
   - `ID`: `"itunes.path-reconcile"`
   - `ResumePolicy`: `ResumeRestart` with phase checkpointing
   - `Phases`: `[{Name: "load_tracks"}, {Name: "match_paths"},
     {Name: "write_results"}]`
   - `Cancellable`: `true`

5. `internal/plugins/itunes/path_repair.go` —
   `pathRepairDef()`:
   - `ID`: `"itunes.path-repair"`
   - `ResumePolicy`: `ResumeAsk` (destructive operation; user picks
     resume vs requeue vs drop after restart)
   - Note: existing code defaults resume to dry-run. Preserve that
     behavior — `ResumeAsk` is the correct policy because the user
     should re-confirm.

6. `internal/plugins/itunes/position_sync.go` —
   `positionSyncDef()`:
   - `ID`: `"itunes.position-sync"`
   - `ResumePolicy`: `ResumeRequeue`
   - `Schedule`: `"*/15 * * * *"` (every 15min) — match existing
     scheduler entry
   - `Capabilities`: `[CapLibraryRead, CapLibraryWrite]`

7. Tests for each def.

## Files to edit

1. `internal/server/server_lifecycle.go` — replace existing
   `case "itunes_import":` (and the others) in
   `resumeInterruptedOperations()` with calls to `EnqueueOp` via the
   registry. After this PR, the old switch cases can be empty;
   they're deleted in UOS-14.
2. `internal/server/itunes_handlers.go` (or wherever the start
   endpoints live) — handlers become thin redirects.
3. `internal/plugins/plugins.go` — import iTunes.
4. `internal/server/server.go` — construct + register iTunes plugin.
5. `internal/server/scheduler_tasks.go` — remove the iTunes
   scheduler entries now that schedules are declared on the
   OperationDefs themselves.

## Hard rules

- `itunes.import` MUST preserve all existing checkpoint behavior;
  test with a real interrupted import → resume → completion.
- `itunes.path-repair` MUST use `ResumeAsk`. Default-to-dry-run on
  resume is implemented by exposing a UI affordance on the
  `interrupted_pending_user` card that re-runs with `dry_run=true`.
- All iTunes ops MUST keep their existing `auth.PermScanTrigger`
  permission requirement.
