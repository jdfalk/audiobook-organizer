## Hard rules for the implementer

1. **No design decisions.** If the spec is silent or ambiguous on a
   specific point, halt and ask. Do not invent.
2. **One task per branch, one PR per task.** Branch name is given in
   each sub-task. Do not mix tasks.
3. **No mixing in unrelated changes.** If a related cleanup is needed
   to make the task pass tests, do it; if a tempting cleanup is
   not strictly needed, skip it.
4. **Versioned file headers required.** Every new or edited file MUST
   have its `// file: ...` / `// version: ...` / `// guid: ...` /
   `// last-edited: ...` header bumped per CLAUDE.md rules.
5. **Tests are mandatory.** Each task names the tests that MUST exist
   before the PR opens. CI must be green.
6. **No `--no-verify`, no skip-CI flags, no force-push to main.**
7. **Use git worktrees per CLAUDE.md.** Branch off `origin/main`,
   present a `PLAN.md` in the worktree before edits, do not commit
   to the primary checkout.

## Verification protocol

After each PR:

1. `make ci` MUST pass locally before opening the PR.
2. CI on the PR MUST pass before requesting review.
3. After merge, deploy to the staging-ish primary checkout via
   `make deploy` and watch journalctl for 5 minutes for unexpected
   errors.
4. If the PR is in Phase A and concerns the canary path, manually
   trigger `embed-scan` on the deployed instance and observe expected
   behavior.
