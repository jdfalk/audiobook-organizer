# Mock CI Gate — MockStore Freshness & Staticcheck

**Status:** Draft — awaiting implementation
**Scope:** `internal/database/mocks/`, `Makefile`, CI pipeline
**Related specs:** none

---

## Problem

`MockStore` (generated by mockery from `Store`) is stale: it does not implement all
methods added to the `Store` interface since the last mockery run. As a result:

- `go vet ./...` fails across 9+ packages because the mock no longer satisfies the
  interface.
- Tests that import the mock cannot compile, breaking the entire test suite.
- `make ci` does not detect this until a developer notices manually.
- `staticcheck` is not run at all, so hundreds of low-hanging static issues go
  unreported.

---

## Core Rule / Goal

> **`make ci` must fail immediately if the mock is stale, and must run staticcheck.**

The mock is considered fresh when `go generate ./...` (re-running mockery) produces
output byte-for-byte identical to the committed `internal/database/mocks/mock_store.go`.

---

## Approach

### MOCK-1 — Regenerate the mock

Run `make generate` (or the mockery command from `docs/MOCKERY_GUIDE.md`) to regenerate
`internal/database/mocks/mock_store.go` against the current `Store` interface.
Commit the result. Verify `go vet ./...` is clean.

### MOCK-2 — Add CI gate

Add two new steps to `make ci` (after the existing build/test steps):

1. **Mock freshness gate:** Re-run `go generate ./internal/database/...`, then
   `git diff --exit-code internal/database/mocks/` — fails if the mock drifted.
2. **Staticcheck:** Run `staticcheck ./...`. Document in the Makefile comment which
   binary to install (`go install honnef.co/go/tools/cmd/staticcheck@latest`).

---

## Acceptance Criteria

- [ ] `go vet ./...` returns zero errors in the main repo.
- [ ] `make ci` runs `go generate` + diff check for mock freshness.
- [ ] `make ci` runs `staticcheck ./...` after `go vet`.
- [ ] Intentionally removing a method from `store.go` causes `make ci` to fail.
- [ ] Adding `staticcheck` to CI does not require modifying workflows — it is in
      `make ci` which is already called by CI.
