## Problem

Target: **80% coverage on the package**. Reaching that requires filling the four gaps and deepening importer/service/transfer/writeback.

## Why this matters

iTunes integration is the highest-stakes write path in the system — it edits the user's iTunes library on a remote Windows machine via .itl binary writes. A regression here corrupts user data. The path-repair and path-reconcile coverage is good *because* we shipped tests aggressively when those were new. The four 0%-coverage files predate the testing discipline; they're the riskiest surface today.

## Goal

For each of the five sub-tasks below, ship tests that:

1. **Cover every exported function** with at least one happy-path case.
2. **Cover error paths** — at minimum: store error, network error (where applicable), context cancellation, malformed input, disabled-mode early-exit.
3. **Cover edge cases** specific to the file (see per-task spec).
4. **Use the existing mock pattern** in the package. Do NOT introduce new mocking frameworks. The package already has `*_mock_test.go` files generated by mockery.

After all five sub-tasks merge, package coverage should exceed **80%**.

## Why split this way

Each file is independently testable. Mixing them defeats the resumability of the burndown bot — if 4.13c fails on flake, 4.13a should still merge.

The splits also follow concern boundaries:

- 4.13a **status.go** — read-only state queries, easy.
- 4.13b **track_provisioner.go** — track-creation logic, narrow.
- 4.13c **validate.go** — pure validation, easiest.
- 4.13d **importer.go** error paths — the big file, hardest split.
- 4.13e **service.go + transfer.go** — lifecycle and remote-write surface.

writeback_batcher.go has a mock test file but real coverage at 58%; deepening that is **4.13f**, lower priority since the file isn't in the 0%-coverage gap list.

## Design decisions

**Subtests, not individual top-level tests.** Use `t.Run("happy_path", func ...)` etc. — keeps the test file scannable and matches the existing pattern in `path_repair_test.go`.

**Real fixtures over hand-built structs.** Where ITL parsing is involved, use the testdata files under `testdata/itunes/` (search for them — they exist).

**Disabled mode is a recurring assertion.** Every entry point should have a `_DisabledModeReturnsEarly` test that builds a service with `Deps.Enabled = false` (or the equivalent disabled flag) and confirms the function early-returns without touching store / network.
